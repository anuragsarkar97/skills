# Product Competitive Thinking

Use this skill before building product-facing work, especially in startups where shipping the wrong thing is more expensive than shipping slowly. Pair it with `critical-thinking`: first ask whether the idea is coherent, then ask whether it wins for users and against competitors.

## Product Manager Lens

1. Identify the target user, job to be done, pain level, and current workaround.
2. State the product bet: acquisition, activation, retention, revenue, expansion, trust, or cost reduction.
3. Define the smallest useful outcome and the metric that would prove progress.
4. Cut scope that does not change user behavior or learning speed.
5. Check whether the feature fits the product narrative, onboarding path, and existing user mental model.

## Competitor Lens

1. Name the likely alternatives: direct competitors, spreadsheets, manual process, internal tools, agencies, or doing nothing.
2. Ask how a competitor would copy, undercut, bundle, or reposition against this feature.
3. Look for defensibility: workflow lock-in, data advantage, speed, trust, distribution, integration depth, or superior UX.
4. Avoid commodity features unless they remove a blocker for a sharper wedge.
5. Do not invent competitor facts. If current market details matter, ask for sources or verify before relying on them.

## Startup Checks

- Does this reduce time to value for a specific user segment?
- Does it create a wedge, or is it a generic checklist feature?
- Is this the smallest experiment that can validate the product bet?
- What does the team learn if this fails?
- Does it increase activation, retention, revenue, or referral potential?
- Is the implementation cost justified by expected learning or strategic leverage?

## Pushback

Say no or propose a smaller experiment when the request is a product anti-pattern:

- Building because competitors have it, without knowing whether users care.
- Adding configuration instead of choosing a strong default.
- Shipping breadth before the core workflow is excellent.
- Optimizing power-user edge cases before activation works.
- Treating internal complexity as product value.
- Copying competitor surface area without matching their distribution or strategy.

## Output

Prefer a compact decision memo:

- Product bet
- Target user and pain
- Competitive alternatives
- Differentiation or commodity risk
- MVP scope
- Metric or learning goal
- Recommendation: build, narrow, defer, or reject
